> 本文是**路线图与目标书**,不含实现、暂不开工。每一层落地时再转成独立的 OpenSpec change

  (+9.4%)、忍者神龟3 369→397(+7.7%)、双截龙3 364→386(+5.8%)**。
  - 取证(`tools/ppu-self-time.sh`,`sample` self-time,非扰动):PPU 逐 dot 核 ≈50–55%、

    (调试钩子已用 `is_empty()`/单 bool 廉价门控)。

## 0. 参考选型 —— 主蓝本 = Mesen2

| 项目 | 强项 | fc 如何用 | 本地路径 |
|---|---|---|---|
| **Mesen2** | cycle/sample 精确、**业界最强调试器** | **主蓝本**(架构+精度判据+调试器) | `Mesen2/Core/NES`、`Mesen2/Core/Debugger` |
| **FCEUmm** | mapper 广度第一(424 board) | mapper 长尾字典 | `libretro-fceumm/src/boards/` |
| **FCEUX** | 按号命名的盗版/华星长尾(173 board)、工具链 | mapper 长尾字典 + 调试/脚本 UX | `fceux/src/boards/`、`fceux/src/debug.cpp` |
| **Nestopia UE** | 边角时序/音频最稳(205 board) | 精度交叉验证 | `nestopia/source/core/board/` |

**为什么是 Mesen2:** fc 的定位是「自研内核 + 调试器 + AI/MCP 为护城河」。四者里只有 Mesen2 在
**精度**与**调试器**两条线都是天花板,其「精度可验证、调试器驱动开发」的哲学与 fc 完全同构。
mapper 纯广度上 FCEUmm/FCEUX/Nestopia 更全,但那是「补长尾的字典」,不是「学架构的蓝本」。

## 1. 现状评估(实测基线,2026-06)

**架构**:cycle-driven lock-step —— `cpu.rd/wr → bus.tick`(PPU×3 + APU×1);PPU 逐 dot `tick()`;
统一 DMA 仲裁器(OAM+DMC,per-cycle)。无逐游戏 hack。底座与 Mesen 同级。

**精度基线(全绿,近期并发开发已大幅推进)**:

| 套件 | 结果 | | 套件 | 结果 |
|---|---|---|---|---|
| cpu_interrupts_v2 | **5/5** | | mmc3_test | **6/6**(含 4-scanline_timing) |
| instr_test-v5 | **16/16** | | sprdma_and_dmc_dma | **2/2**(DMA 逐拍已校准) |
| apu_test | **8/8** | | ppu_vbl_nmi | **10/10** |

**冻结回归门已从 47 扩到 56 个 ROM**(`baseline-extended.json`),新纳入 9 个已实测全绿的 $6000-协议套件:
`oam_read`、`oam_stress`、`ppu_open_bus`、`cpu_dummy_writes`(oam/ppumem)、`cpu_exec_space`(apu/ppuio)、
`instr_misc`、`instr_timing`。覆盖 OAM 读/压力、PPU open-bus、CPU 双写、CPU 在 I/O 空间执行、指令时序。

> **待攻复核(2026-06,逐屏渲染核对)**:原列「待攻」的 **`sprite_overflow_tests` 5/5 全过**
> (Basics/Details/Timing/Obscure/Emulation),`mmc3_irq_tests`(Disch)、`cpu_timing_test6`、
> `cpu_dummy_reads` 也都过——它们此前被 `$6000` 打分器误判为 TIMEOUT,实为非 $6000 协议(屏显/串口),
> 渲染核对均 PASS。**真正仍待攻的只剩 `dmc_dma_during_read4`**(屏显校验和协议 + DMC 逐拍精同步,
> 需参考校验和),及与 Mesen 的逐拍 trace parity(已用 `nestest.log` 金标准做了 5003 条恒定偏移、
> 零漂移的逐拍 parity,见 `tools/nestest-parity.py`)。

**mapper(15 个)**:0,1,2,3,4,5,7,9,10,11,25(VRC4),66,71,74,194。覆盖主流;**长尾缺口大**
(无 VRC6/7、N163、FME-7/5B、FDS,及绝大多数盗版多合一与华星扩展号)。

**调试能力(现有)**:exec/read/write 断点 · **条件断点(✅ `expr.rs`)** · 单步指令 · 内存读写 ·
反汇编 · **trace logger(✅ `fc trace`/`emu_trace`)** · pattern/nametable 查看 · PPU/APU 状态 ·
源码行↔地址映射(cc65 `.dbg`)· **MCP `emu_*` 工具**(独有,含条件断点/trace)。
**缺口**:event viewer、实时 OAM/CHR/调色板查看器、访问热力图、rewind、call stack、
step over/out、profiler。

**性能基线(实测)**:release headless SMB **≈407 fps**(6.8× 实时,见上方进度快照);`tauri dev`
已修为 `opt-level 3`,动作游戏实测 emulation 稳 ~60fps。距 §2 目标(≥2500fps)约 6×,空间很大。

## 3. 分层详细计划(L0 → L4,逐层递进、各自可交付)

### Layer 0 —— 测量与对拍地基 ★必须先做(后面全部依赖)

**目标**:让任何优化都「可测、可回归、可对照 Mesen」。产出三件工具 + 一份基线。

**任务**
- **0.1 测试矩阵自动化**:`fc testsuite --json`,把 `nes-test-roms` 全套按 suite 跑成
  PASS/FAIL/TIMEOUT 表;非 `$6000` 协议的(sprite_overflow、dmc_dma_during_read4)改走**读屏/串口
  CRC 打分**而非误判 TIMEOUT。落地为一个回归脚本 + 基线 JSON。
- **0.2 性能基准**:`fc bench` —— headless 跑 N 帧,输出总 fps + **子系统耗时占比**(CPU/PPU/APU/
  mapper),固定三档场景:SMB(轻)/ 忍者神龟(重精灵)/ MMC3 卷动(重 A12)。
- **0.3 Trace 对拍工具 ★精度核心武器**:`fc trace` —— 逐指令(可选逐周期)日志,**格式对齐
  Mesen / Nintendulator** 的 trace log;配 `tools/trace-diff` 与 Mesen 输出做 diff,定位首个分歧点。
- **0.4 基线冻结**:把当前 6 套件全绿 + 730fps 记入基线,作为「不得回归」红线。

**验收**:三件工具可用;一次 `fc trace` 能和 Mesen 的 SMB trace 对齐到同一格式并 diff。

### Layer 1 —— 性能(快赢 → 架构,保持「加精度不掉帧」的余量)

**目标(量化)**:release headless ≥2500fps(SMB)/≥1200fps(重场景去精灵限);GUI 最重场景
单帧 emulation P99 < 8ms;**调试钩子开关 fps 差 ≤5%**;精度矩阵零回归。

**任务**
- **1.1 PPU 逐 dot 降密度**:把 `tick()` 里的 `if self.dot == X` 长链改为**分段状态机/查表**
  (可见区/取数阶段/HBlank/VBlank 分段处理),对照 Mesen `NesPpu.cpp` 的 `ProcessScanline`/
  `LoadTileInfo`/`LoadSprite*` 分段写法。基准:重场景 PPU 占比下降。
- **1.2 render_pixel 去拷贝**:消除每像素的 `SpriteUnit` 结构体 copy(256×240×逐精灵);改用索引/引用。
- **1.3 帧缓冲直出 u32**:渲染直接写 palette 预乘的 `u32`(LUT),省掉前端 RGBA 转换;复用 0.28ms pollFrame。
- **1.4 双路径(★关键)**:把 watchpoint / trace / access-counter 等调试钩子从核心热路径剥离,
  用运行期开关:**不调试时热路径零调试分支**(对照 Mesen 的 `Debugger` 旁路注入)。这是「精度与
  性能不打架」的总开关 —— L3/L4 加的精度与采样成本只在调试模式付费。
- **1.5 APU 重采样**:blip-buf 风格的带限重采样(对照 `NesSoundMixer.cpp`),降低每样本成本。
- **1.6 编译/链接**:确认 release `lto=true, codegen-units=1`;dev 维持 `opt-level 3`(已做)。

**参考**:`Mesen2/Core/NES/{NesPpu.cpp, NesSoundMixer.cpp}`、Mesen 的 Debugger 旁路设计。
**验收**:`fc bench` 前后对比表达标;6 套件零回归;调试开关 fps 差 ≤5%。

### Layer 3 —— 模拟精度(逐拍对齐 Mesen)

**目标**:绿项零回归;待攻转绿;与 Mesen 三档游戏 trace 前 10 万条指令 0 diff;APU 非线性混音对齐。

**任务**
- **3.1 Trace parity(主线)**:用 L0 的 `fc trace`,对 SMB / 一款动作 / 一款 RPG 与 Mesen trace
  逐条 diff,定位首个分歧→修→重跑,直到 0 diff 收敛。
- **3.2 PPU 边角**:sprite overflow 精确时序、OAMADDR 破坏、`$2004` 读、open-bus
  (`OpenBusHandler.h`)、odd-frame skip、读 `$2002` 抑制 NMI 的边界。
- **3.3 APU**:DMC 时序边角、frame counter 5-step/4-step 边界、**非线性混音表**对齐
  `NesSoundMixer.cpp`、相位/复位。
- **3.4 系统级**:上电/复位寄存器状态、open-bus decay、controller 连读、PAL/Dendy 分数时钟。
- **3.5 dmc_dma_during_read4**:DMC 精同步收敛(配 L0 的读屏/串口打分)。

**参考**:`Mesen2/Core/NES/{NesPpu.cpp, NesSoundMixer.cpp, OpenBusHandler.h}`、Nestopia 对照。
**验收**:每项对应 test rom 绿;Mesen trace diff 报告(0 diff 段长);音频频谱对照通过。

### Layer 4 —— 断点调试能力 ★护城河(对标 Mesen2/Core/Debugger,并用 MCP 放大)

**目标**:fc 调试器达 Mesen 功能子集,且**全部经 MCP 暴露**,使 **AI agent 能自主完成一次 bug
定位闭环**(这是 fc 超越 Mesen 的方向 —— Mesen 的强大调试器只有人能用,fc 让模型也能用)。

**任务(逐组件移植,左 fc 目标 / 右 Mesen 参考)**
- **4.1 条件断点**:表达式(A/X/Y/P/SP/PC/cycle/scanline/dot/读写值/地址),按内存类型(CPU/PPU/OAM)
  ← `BreakpointManager.cpp` + `Breakpoint.cpp` + `ExpressionEvaluator.cpp` + **`ExpressionEvaluator.Nes.cpp`**。
- **4.2 Trace Logger**:可配置格式 + 条件触发,环形缓冲/落盘 ← `BaseTraceLogger.h` +
  `NES/Debugger/NesTraceLogger.cpp`。MCP:`emu_trace`。
- **4.3 Event Viewer**:scanline×dot 网格标注寄存器写/NMI/IRQ/sprite0/DMA ← `BaseEventManager.cpp` +
  `NES/Debugger/NesEventManager.cpp`。前端画布 + MCP:`emu_event_dump`。
- **4.4 实时查看器 + 访问热力图**:Nametable/CHR tile/Sprite(OAM)/Palette,read/write/exec 着色 ←
  `PpuTools.cpp` + `NES/Debugger/NesPpuTools.cpp` + `MemoryAccessCounter.cpp` + `CodeDataLogger.cpp`。
  MCP:`emu_view_nt / emu_view_chr / emu_view_oam / emu_heatmap`。
- **4.5 执行控制**:step into/over/out(`DummyNesCpu.cpp`)、step scanline/frame、run-to NMI/IRQ/scanline
  ← `Debugger.cpp` 的 StepRequest。
- **4.6 Rewind**:状态环缓冲 ← `Shared/RewindManager.cpp` + `RewindData.cpp`。MCP:`emu_rewind`。

## 8. 风险与纪律
- **精度↔性能**张力 → 用 L1.4 双路径化解(默认快,调试/验证才开精确成本)。
- **mapper 长尾无止境** → 只补「用户库实际出现 + 经典必备」,不追全量。
- **不引入逐游戏 hack**;**不为过单个 test rom 加固定 tick**(沿用 `DMA仲裁设计说明.md` 的纪律)。
- **save-state 结构**随精度/调试字段演进,不为旧档兼容牺牲正确性。
- 与**并发开发**协调:近期内核被活跃修改(已修 instr 16/16、sprdma 2/2、新增 mapper 194);
  各 change 前先 `git pull`/对齐,避免冲突。
