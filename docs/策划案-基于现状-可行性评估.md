# 复古 FC 一体化 IDE —— 基于现状的策划案与可行性评估

> 本文是对《复古FC游戏集成开发策划案.md》愿景的**落地版**:以当前代码库的真实能力为基线,
> 逐模块评估可行性(工作量 / 难度 / 风险),给出战略定位与重新分期的落地路线。
> 结论先行:**模拟器 + 调试器 + AI/MCP 这条线是真护城河且可行;一整套"零依赖自研"创作工具链是多人月级别的工程,需要重新取舍。**

---

## 二、逐模块可行性评估

工作量按**单人/小团队**估算(人周 = 1 人 5 天),以"现有模拟器 4.3k 行"为难度参照系。

| 模块 | 还需 | 工作量 | 风险 |
|---|---|---|---|
| 模拟器内核 | 精度长尾(子周期时序)、更多 Mapper | 持续 | 低(已可玩) |

| ROM 打包/头编辑 | 可视化 iNES 头 + 拼 PRG/CHR(逻辑已会读) | 0.5–1 人周 | 低 |
| 十六进制编辑器 | 复用调试器内存视图组件 | 1 人周 | 低 |
| CHR 图形编辑器 | Canvas 画 8×8 图块 + 4 色调色板 + 导出 .chr/.inc | 2–3 人周 | 中 |
| 地图编辑器 | 图块拼接 + 碰撞层 + 导出字节数组 | 2–3 人周 | 中 |

### C. 高风险/高成本项(需重新取舍)
| 模块 | 真相 | 自研成本 | 建议 |
|---|---|---|---|
| **6502 汇编器/编译器** | cc65 是几十年积累的 C 编译器+汇编器+链接器;"自研双语法全兼容"不现实 | 可用子集 4–8 人周;cc65 全兼容 ≈ 不可达 | **先封装 ca65/ld65 作为 Tauri sidecar**(1 人周打通,零兼容风险);自研汇编器作为后期差异化,只做 asm6 级最小可用子集 |
| **2A03 音频编辑器** | Famitracker 是 20 年的专业 tracker,效果器/乐器/包络极深 | 基础 tracker(5 声道+音序+导出)4–6 人周;对标 Famitracker 6–12 人月 | **降级**:先做"基础 tracker + 导入 .nsf/.ftm 回放",对标 Famitracker 留到后期或不做 |

### D. AI / MCP(最强差异化,且最可行)
| 层 | 现状 | 工作量 | 说明 |
|---|---|---|---|
| 硬件层 MCP | ✅ 已有(fc-mcp) | —— | 唯一性来源:AI 能真读 PPU/内存/断点 |
| 工程层 MCP | ❌ | 2–3 人周 | 暴露工程文件/源码/编译日志/打包配置;**复用 fc-mcp 的 stdio JSON-RPC 模式即可** |
| AI 智能体 | ❌ | 2–4 人周 plumbing | 接线易,**智能体质量靠大模型本身**;6502 编码/排错/硬件调试三个最值得先做 |
| 开发期 AI 调试 | ✅ 已打通 `fc tauri-bridge`(eval JS 调真实界面) | —— | 这条已能用,见 `.mcp.json` |

## 三、可行性总判

1. **底座(模拟器+调试器+硬件 MCP)= 真护城河,已基本就位。** 这是同类"AI 玩/调游戏"项目(mcp-gameboy/lmgame)都没有的精度向调试能力。
2. **"完全自研、零第三方依赖"的全套创作工具链 = 12–24 人月+ 的工程**,单人短期不现实。最贵的两块是**汇编器**(对标 cc65)和**音频编辑器**(对标 Famitracker)。
3. **可行的破局点**:不要既做完美播放器又做完整 IDE。**先把"工程 → 编辑 → 汇编(封装 sidecar)→ 运行 → AI 硬件级调试"这条最小创作闭环打通**,这正好把已有的模拟器+调试器+MCP 全部变现,且是别人做不出的差异化。
4. **战略取舍建议**:`自研无依赖` 是个**成本极高的营销点**,不应成为 V1 的硬约束。务实路线 = **关键卡脖子环节(汇编/音频)先封装成熟工具,把自研预算押在护城河(模拟器/调试/AI-MCP)上**;待产品验证后再决定是否自研替换。

## 四、重新分期的落地路线(务实版)

> 原文档 V0.1–V0.4 偏乐观。下面按"每阶段都能交付可用产品"重排,并标注是否已具备。

### M1 · 创作最小闭环(MVP,~6–8 人周)
目标:**能新建工程 → 写汇编 → 一键汇编 → 内嵌运行 → 调试**。
- 工程管理 + 文件树 + 2 个模板(2–3 人周)
- CodeMirror + 6502 高亮/补全(1.5–2 人周)
- **封装 ca65/ld65 sidecar** + 编译日志/报错跳转(1.5 人周)
- ROM 头可视化打包(0.5–1 人周)
- 复用现有模拟器内嵌 + 调试器(✅ 已有)
- 编译报错跳转 ↔ 代码行、断点 ↔ 代码行(1 人周)

### M2 · 资源工具 + AI 排错(~6–8 人周)
- CHR 图形编辑器(2–3 人周)
- 地图编辑器(2–3 人周)
- **工程层 MCP** + AI 编译排错 + 6502 编码助手(3–4 人周,复用 MCP)

### M3 · 深度调试 + 音频(~8–10 人周)
- PPU 显存预览/精灵列表、运行轨迹、错误快照(2–3 人周)
- 基础 2A03 tracker(4–6 人周)或先做 .ftm/.nsf 回放
- AI 硬件级调试智能体(读快照→区分代码/资源/硬件 Bug)(2–3 人周)

### M4 · 打磨与生态
- 自研 6502 汇编器(可选,替换 sidecar)、更多 Mapper、FDS
- 多智能体调度、权限隔离、自定义智能体接入文档
- 新手引导、案例库、教程

## 六、风险重评(在原文档基础上修正)

- **最大风险不是技术,是范围**:全自研全工具链 = 范围爆炸。**对策:M1 最小闭环验证产品价值,再决定投入。**
- 汇编器自研:cc65 兼容是无底洞 → **封装规避**。
- 音频编辑器:Famitracker 对标不现实 → **降级**。
- AI 生成不符合 FC 规范:**已有硬件规则可做校验引擎**(模拟器内核就是 ground truth,可让 AI 改完直接在内核里跑测试验证 —— 这是别人没有的闭环优势)。
- 截屏调试依赖 macOS 录屏权限(`tauri_screenshot` 当前被系统权限挡;`tauri_eval` 不受限已可用)。

---

## 七、结论

- **可行且应立刻做**:M1 创作最小闭环(工程+编辑+sidecar 汇编+运行+调试),把已有的模拟器/调试器/MCP 变现,~6–8 人周即可拿出"能写能跑能调"的 IDE 雏形。
- **可行但需降级/封装**:CHR/地图/音频/汇编器 —— 用封装或最小子集换时间。
- **最强差异化、性价比最高**:AI + MCP 的"硬件级调试 + 内核即校验"闭环 —— 这是同赛道唯一,且基础设施已就绪,应重点投入。
- **不建议**:把"完全自研零依赖全工具链"当 V1 硬指标,会拖死项目。

> 一句话:**护城河已经挖好(模拟器+调试+AI/MCP),现在缺的是把它包成"创作闭环";卡脖子的两块(汇编/音频)先借力,别自己造轮子。**

# 【战略 v2 · 终版】集成优先:复用开源,做一栈式 AI-native 工作流

> 决策已定:**不再追求"完全自研零依赖"。产品价值 = 一栈式集成 + 唯一的自研模拟器/调试器/AI-MCP。**
> 我们不重造 cc65 / Famitracker / 编辑器内核 —— 我们把业界最好的开源工具**无缝粘合**成一个 AI 原生的统一工作流。
> 这把整体可行性从 **12–24 人月** 降到 **~3–4 人月**,且差异化更清晰。

## v2-1 模块 → 开源工具集成图

| 模块 | 复用的开源工具 | 许可证 | 集成方式 | 自研 vs 集成 |
|---|---|---|---|---|
| 汇编/链接/C 编译 | **cc65(ca65 + ld65 + cc65)** | zlib(可商用闭源打包) | **Tauri sidecar**,headless CLI 调用;捆绑 `.cfg` 链接脚本 + 设 `CC65_INC/LD65_LIB/LD65_CFG` | 集成 ✅ |
| 代码编辑器 | **CodeMirror 6 / Monaco** + 现成 6502/ca65 TextMate 语法 | MIT | 内嵌 webview,**无缝** | 集成 ✅ |
| 音频编辑 + 引擎 | **FamiStudio**(编辑 + CLI 导出)+ FamiStudio/FamiTone2 声音引擎(ca65) | MIT | CLI 导出 `.fms → .s/.nsf`(`-famitone2-format:ca65`);引擎 asm 作为工程模板;编辑用其 GUI(launch+watch)或内置轻量 tracker | 集成 ✅(原最难项被消灭) |
| CHR 图形 | 内置 Canvas 轻量编辑器(8×8/4 色)/ 可选对接 NEXXT | — | 内置自研(简单)或外部工具按格式对接 | 轻自研 🔶 |
| 地图/命名表 | **Tiled** / NEXXT,或内置轻量拼图 | BSD/GPL(注意发行) | 按文件格式对接 + 转换器;或内置 | 轻自研/集成 🔶 |
| ROM 打包/头编辑 | 内置(逻辑已会读 iNES) | — | 内置 | 自研(易)✅ |
| 模拟器/调试器 | **本项目自研(护城河)** | — | 内嵌运行 + 调试器(已有) | 自研 ✅ |
| AI / MCP | **本项目自研 fc-mcp + 工程层 MCP** | — | 驱动整条工具链 | 自研 ✅ |

> 关键:`cc65` 和 `FamiStudio` 都是 NES homebrew 社区**事实标准**,生态/教程/案例现成,用户无学习迁移成本。

## v2-2 护城河重定义(为什么别人做不出)

集成本身不是壁垒(谁都能装 cc65)。**真壁垒 = 自研模拟器/调试器/AI-MCP 这一层,把集成升维成"AI 原生 + 硬件级"的闭环:**

1. **内核即校验(独家)**:AI 改完代码/资源,直接在**自研模拟器内核**里跑、读 PPU/内存断言对不对 —— 别的 IDE 没有可编程驱动的内核做 ground truth。
2. **硬件级 AI 调试(独家)**:fc-mcp 已能让 AI 读寄存器/内存/PPU/APU、下断点、抓快照。Mesen 有调试器但无 AI/集成;cc65 无 IDE;FamiStudio 只管音频 —— **没有任何一家把"集成 + 调试器 + AI"三者合一**。
3. **一栈式 orchestration**:统一工程模型 + 一键"编译(ca65)→ 链接(ld65)→ 合资源 → 打包 .nes → 内嵌运行 → 调试",AI 可端到端编排整条流水线。

> 定位一句话:**"NES homebrew 的 AI-native 一栈式工作台"** —— 集成现成最佳工具,独家提供"可编程内核 + 硬件级调试 + AI 编排"。

## v2-3 集成架构(新增的工程重点)

```
前端(Vue3 IDE 外壳:文件树 / 编辑器 / 资源面板 / 调试面板 / AI 面板)
        │ Tauri IPC
Rust 编排层(本项目核心新增):
   · 统一工程模型(project.toml:源码/CHR/音乐/地图/Mapper/链接配置)
   · 构建编排器:ca65 → ld65 → 拼 .nes;FamiStudio CLI 导出音乐 .s
   · 文件监听 + 增量构建;sidecar 进程管理;日志解析(报错跳转)
   · 工程层 MCP:把"工程上下文 + 构建 + 运行"暴露给 AI
        │
   sidecars: ca65 / ld65 / (cc65) / FamiStudio(CLI 导出)
   内嵌:自研模拟器 + 调试器(已有)
```

**集成工作的真重点(取代了"造编译器"):**
- **跨平台二进制打包**:ca65/ld65(小)+ FamiStudio(.NET 8,需 self-contained ≈ 50–100MB)按 target-triple 打包;macOS 需对捆绑二进制做签名/公证。
- **统一工程模型 + 构建编排器**:把各工具的输入/输出粘成一条可重放的构建流水线(这是"一栈式"的技术内核)。
- **格式胶水**:`.fms`→`.s`、图片→CHR、Tiled→字节数组 的转换器。
- **UX 无缝度**:80% 常用编辑做成**内置轻量编辑器**;重度场景"在 FamiStudio/Tiled 中打开 + 文件回写同步";避免"像在拼几个独立软件"。

## v2-5 落地路线(集成版,更快)

- **M1 创作闭环(~4–5 人周)**:工程模型 + 文件树 + CodeMirror/6502 + **cc65 sidecar 一键编译** + ROM 打包 + 内嵌运行/调试(已有)+ 报错/断点 ↔ 代码行。
- **M2 资源 + 音频(~4–6 人周)**:内置 CHR/地图轻量编辑器 + **集成 FamiStudio(CLI 导出 + 引擎模板)** + 图片/地图转换器。
- **M3 AI 一栈式(~4–6 人周)**:工程层 MCP + AI 编排(新建工程/编码/编译排错/硬件调试)+ 内核即校验闭环 + AI 面板。
- **M4 打磨/生态**:跨平台打包签名、模板/案例库、"在外部 pro 工具打开"、更多 Mapper/FDS、自定义智能体接入。

## v2-6 集成专属风险

| 风险 | 对策 |
|---|---|
| FamiStudio 是 .NET,捆绑体积大 | self-contained 单文件发布;或可选下载;或仅捆绑其 CLI 导出所需部分 |
| 跨平台二进制 + macOS 公证 | CI 预编译 ca65/ld65;捆绑二进制纳入签名/公证流程 |
| 多工具版本漂移 | 锁定版本号、内置;提供"升级工具链"入口 |
| UX 像拼装而非一体 | 80% 内置轻量编辑;重度"外部打开+回写";统一构建/运行/调试入口 |
| 许可证合规 | 捆绑各工具 LICENSE(zlib/MIT),"第三方许可"页;不冒认作者 |

## v2-7 结论(终版)

- **战略正确**:集成优先把项目从"多人月造轮子"变成"~3–4 人月做粘合 + 护城河变现",且差异化(AI 原生 + 硬件级调试 + 内核即校验)反而更突出。
- **现成可借力且许可证友好**:cc65(zlib)、FamiStudio(MIT)、CodeMirror/Monaco(MIT)—— 全部允许商业闭源打包。
- **自研只留在护城河**:模拟器、调试器、构建编排器、工程层 MCP、AI 编排。
- **立即可启动**:M1 创作闭环以 cc65 sidecar 为核心,~4–5 人周即可拿出"能写、能编、能跑、能调"的一栈式雏形。

# 附录 A · 音频模块专项评估与策略(终定)

> 针对"是否重写 FamiStudio"的专项评估与最终取舍。
> **结论:本地精简自研一个聚焦 2A03 的内置 tracker(覆盖 80%);专业需求不集成 FamiStudio,而是引导用户自行下载、提供"导出物对接回写本工程"的能力与规范。最大化省工作量。**

## A-1 重写难度评估(为什么不全量对标)

"重写 FamiStudio"有两种含义,难度差一个数量级:

| 方案 | 含义 | 工作量 | 取舍 |
|---|---|---|---|
| **A 全量对标** | 6 颗扩展芯片 + 钢琴卷帘 + NSF/FTM/MIDI 导入 + 视频导出 + 移动端 | **数人年** | ❌ 不做 |
| **B 内置精简 tracker** | 2A03 五声道 + 包络乐器 + DPCM + 音序/卷帘 + 汇编导出 + 基础导入 | **~3–5 人月** | ✅ 做 |

FamiStudio 是单人十几年积累(基于 Shiru 的 FamiTone2),C#+C++ 几万行;全量重写既不划算也无必要。

## A-2 起跑线优势:核心已具备

NES 音乐工具最难的"正确性内核 = 2A03 声音合成,本项目已有**:`fc-core/apu.rs`(Pulse×2 / Triangle / Noise / DMC / 帧序列器 / 混音+DC-block,周期级,~700 行)。FamiStudio 的播放保真靠其 `NesSndEmu`(C++ DLL)—— **我们用自研内核替代,且 `cpal` 输出现成**。
> 闭环优势:tracker 播放 = "音符+乐器+效果 → 每帧写 APU 寄存器"。**用同一台 APU 内核做预览播放**,AI 还能在内核里"听"和断言,这是 FamiStudio 没有的。

## A-3 难度逐项(本栈:Rust 内核 + Tauri/Vue)

| 模块 | 难度 | 工作量 |
|---|---|---|
| 2A03 合成 | — | ✅ 已有 |
| 播放引擎(音符/乐器/效果→寄存器,逐帧,驱动现有 APU) | 中 | 2–3 人周 |
| 乐器系统(音量/琶音/音高/占空比包络 + DPCM 映射/采样) | 中 | 2–3 人周 |
| 工程模型 + 撤销/重做 | 低中 | 1–2 人周 |
| 音序 / Pattern 视图(Vue/Canvas) | 中 | 2–3 人周 |
| **钢琴卷帘编辑器(招牌 UX,重度交互 Canvas)** | **高** | **4–8 人周** |
| 效果(滑音/颤音/琶音/音量轨) | 中高 | 2–4 人周 |
| 汇编导出(对齐 FamiStudio/FamiTone2 引擎格式) | 中高 | 2–4 人周 |
| 基础导入(FTM 文本)| 中 | 2–4 人周 |

## A-4 三道"悬崖"(全量对标的成本元凶,B 方案全部回避)
1. **扩展音频芯片**:VRC6、**VRC7(OPLL FM)**、FDS、MMC5、**N163(8 波表声道)**、Sunsoft 5B(AY PSG)—— 每颗独立合成芯片,光合成 ~10–15 人周。**B 方案不做,留给 FamiStudio。**
2. **钢琴卷帘人机工程**:做到顺手极磨人。**B 方案做"够用"版本即可。**
3. **NSF / FTM 二进制导入保真**:出了名的难。**B 方案不做,改为对接 FamiStudio 导出物。**

## A-5 终选策略:内置精简 + FamiStudio 外部可选(不集成)

```
日常 80%:  本工程内置精简 tracker(2A03,驱动自研 APU)—— 无依赖、体积小、AI 深联动
专业/扩展: 用户自行下载 FamiStudio(不打包)→ 在其中创作 → 导出汇编 → 落入本工程目录 → 自动纳入构建
```

**为什么不打包 FamiStudio(省工作量的关键):** 打包要解决跨平台 .NET 自包含发布(~50–100MB)、各平台二进制、macOS 签名公证、版本漂移 —— 这些**全部省掉**。改为**纯文件对接**,我们只负责"接得住、回写得进、能试听"。

## A-6 对接回写机制(具体怎么做,低成本)

方向:**FamiStudio → 本工程**(主路径,够用)。不做双向同步(改回 `.fms` 成本高、收益低)。

1. **工程约定**:工程内 `music/` 目录存放 FamiStudio 导出的**声音引擎 + 乐曲数据汇编**(ca65 格式)。`project.toml` 登记乐曲文件与曲目名。
2. **导出规范(提供文档 + 一键校验)**:固化推荐的 FamiStudio 导出配置 —— *Sound Engine = FamiStudio*(或 FamiTone2)、*Format = CA65*、单/多文件、`{project}_{song}` 命名模板;DPCM 同名 `.dmc` 一并落盘。用户照做即可无缝。
3. **"导入 FamiStudio 导出"入口**:UI 一个按钮,选中导出的 `.s`(及 `.dmc`)→ 拷入 `music/` → 登记进工程。
4. **文件监听 + 自动重建**:`music/` 变化触发构建编排器重新 `ld65` 链接进 `.nes`,**改完音乐即可在内嵌模拟器里听到**(回写闭环)。
5. **试听/校验**:用**自研 APU + 模拟器内核**直接跑导出的 NSF / 链接后的 ROM 试听;AI 可读 APU 通道电平做"爆音/缺声"校验。
6. **(可选,后期)轻量解析**:解析导出 asm 的标签或 FamiStudio 文本导出,在 UI 里展示曲目/乐器清单,不做编辑。

> 一句话:**我们不当 FamiStudio 的宿主,只当它的"下游集成点"** —— 接住它的导出、塞进构建、用自家内核试听校验。

## A-7 工作量(本策略,已最大化压缩)

| 项 | 工作量 | 说明 |
|---|---|---|
| 内置精简 tracker(B,主体) | 3–5 人月 | 自研主体;可按 M 阶段分批(先音序+乐器+导出,卷帘后补) |
| FamiStudio **不打包** | **省 1–2 人周** | 免掉 .NET 跨平台/签名 |
| FamiStudio **对接回写** | **0.5–1 人周** | 导出规范文档 + 导入按钮 + `music/` 监听 + 构建纳入 |
| 内核试听/AI 校验 | 复用已有 | APU/MCP 现成 |

**净效果**:专业音频这条线从"对标 6–12 人月 / 打包数人周"压到**"对接 ~1 人周"**;创作主体只保留一个**够用的内置 2A03 tracker**(可后置、可分批)。符合 v2"护城河自研、卡脖子借力、最大化省工作量"的总基调。
