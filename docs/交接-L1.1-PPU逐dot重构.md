# 交接说明 · L1.1 —— PPU 逐 dot 重构(性能专项)

> 给「新开对话专项推进」用的自包含背景。读完这份 + `docs/模拟器优化计划.md` 即可独立开工。

## 一句话任务

把 `fc-core/src/ppu.rs` 的 `Ppu::tick()`(逐 dot 的长 `if dot == X` 分支链)重构成**分段状态机**,
**显著降低单 dot 成本**,在**不改变任何模拟时序行为**的前提下提升 headless fps。对标 Mesen2
`NesPpu.cpp` 的 `ProcessScanline`/`LoadTileInfo`/`LoadSprite*` 分段写法。

---

## 项目背景(最少必要)

`fc` 是从零自研的 cycle-driven NES/Famicom 模拟器**内核**(Rust workspace),四前端 + MCP 层。
- **核心不变量(lock-step clocking)**:`cpu.rd/wr`(`cpu.rs`)在每次访存**前**调用 `bus.tick()`
  (`bus.rs`),每个 tick 推进 **PPU 3 dot + APU 1 cycle**(+ DMC DMA)。CPU/PPU/APU 子指令级同步,
  **无逐游戏 hack**。改动读/写/访问顺序就会改时序——必须谨慎。
- 所有前端只经 `control_deck.rs` 的 `ControlDeck` 驱动机器。
- 工作区成员:`fc-core`(纯逻辑,无 IO)、`fc-cli`(`fc` 二进制)、`fc-mcp`、`fc-gui`;`fc-tauri` 独立构建。

## 当前性能状态(2026-06,release headless,`fc bench --profile` 实测)

| 场景 | fps | PPU 渲染输出 | APU 重采样 | **其余 = CPU + PPU逐dot核 + mapper** |
|---|---|---|---|---|
| SMB(轻) | ~416 | 6.3% | 3.1% | **90.6%** |
| 忍者神龟3(重精灵) | ~376 | 6.8% | 2.5% | **90.6%** |
| 双截龙3(MMC3) | ~382 | 7.3% | 1.5% | **91.1%** |

**结论**:APU、渲染输出都已不是瓶颈。**~90% 的帧时间全在「其余」**,其中 **PPU 的 `tick()` 是头号
嫌疑**——它每个 CPU cycle 被调 3 次(~89342 dot/帧),里面是密集的逐 dot 分支。这就是本专项要攻的点。

> `fc bench --profile` 的归因法 = 「消融」:跑全量,再跑各「只跳过某子系统*输出*工作、不改模拟行为」
> 的变体,用 fps 差归因(见 `fc-cli/src/main.rs` 的 `Commands::Bench` 与 `ControlDeck::set_profile_ablation`)。
> 注意:目前**还无法把「其余」拆成 CPU vs PPU核**(两者 lock-step 交织,消融会破坏 NMI/sprite0 行为)。
> 专项第一步可考虑:加一个**只跳过 PPU 取数/移位/sprite 评估、但保留 dot 推进 + VBL/NMI + sprite0**
> 的消融档,以把「其余」里 PPU 核的占比单独量出来(谨慎,别破坏 sprite0 依赖)。

## 当前 PPU 代码结构(动手前先读)

文件:`fc-core/src/ppu.rs`(~1247 行)。关键:
- `Ppu::tick(&mut self, cart)` 从 **第 328 行**起——逐 dot 推进。形态(节选):
  ```rust
  pub fn tick(&mut self, cart: &mut Cartridge) {
      self.master_cycle += 1;
      self.clock_nmi_delay();
      let visible = self.scanline < 240;
      let prerender = self.scanline == 261;
      if prerender && self.dot == 1 { /* clear vbl/sprite0/overflow */ }
      if self.rendering() && (visible || prerender) {
          if visible && self.dot == 65 { self.schedule_sprite_overflow(); }
          if (self.dot >= 2 && self.dot <= 257) || (self.dot >= 321 && self.dot <= 337) {
              self.update_shifters();
              match (self.dot - 1) % 8 { 0 => …nt fetch, 2 => …attr, 3 => …bg_lo, 5 => …bg_hi, 7 => inc_x, _ => {} }
          }
          if self.dot == 256 { self.increment_scroll_y(); }
          // … dot==257 copy_x、280..=304 copy_y、sprite 评估、odd-frame skip、VBL set@(241,1) 等
      }
      // … render_pixel()(可见区)、NMI set/clear、scanline/dot 推进与回绕 …
  }
  ```
  即:大量 `if self.dot == X` / 区间判断,每 dot 都全部走一遍。**这是 L1.1 要消除的分支密度。**
- 相关子函数:`render_pixel`(第 ~745 行,逐像素输出)、`update_shifters`、`load_bg_shifters`、
  `increment_scroll_x/y`、`copy_*`、`evaluate_sprites`/sprite 评估、`schedule_sprite_overflow`、
  `update_nmi`/`clock_nmi_delay`、`ppu_read`/`ppu_read_for`(经 cart,**触发 MMC3 A12 / MMC2/4 latch**——
  顺序敏感!)。
- 已存在的 profiler 钩子:`Ppu::profile_no_output: bool`(消融渲染输出)、`Apu::profile_no_resample`,
  经 `ControlDeck::set_profile_ablation(no_render, no_apu)` 暴露,`fc bench --profile` 使用。

## 硬约束 / 纪律(来自 `CLAUDE.md` 与 `docs/模拟器优化计划.md` §8)

1. **lock-step 不变量神圣**:重构只能改「同一 dot 内做哪些工作、怎么组织分支」,**不得改变**任何
   读/写/访问的**相对顺序与时点**(VBL set/clear、sprite0 hit、sprite overflow、odd-frame skip、
   A12 边沿、`$2002` 读抑制 NMI 的边界都对 dot 精确敏感)。
2. **不引入逐游戏 hack**;**不为过单个 test rom 加固定 tick**。
3. `fc-core` 保持无 IO。
4. **save-state 兼容**:若给 PPU 加/改字段,注意 `#[serde]`;不为旧档牺牲正确性,但别无谓破坏。
5. **并发开发**:内核近期被另一会话活跃修改(已修 instr/sprdma、新增 mapper 等)。**开工前 `git status`/
   对齐,改 `ppu.rs` 前重读该文件**(可能已变),小步提交避免冲突。

## 验证方法论(三道门,每改一版都跑)

1. **精度基线门**(最硬):`testsuite --baseline …`,47/47 不得变红;尤其 `ppu_vbl_nmi`、`sprdma`、`mmc3`。
2. **性能门**:`fc bench --profile` 重构前/后对照,「其余」占比 / 总 fps 应改善且无其它项恶化。
3. **Trace parity(最敏感,强烈建议)**:`fc trace`(nestest/Nintendulator 格式)对 SMB 跑前 N 万条,
   与重构前自身或 Mesen 的 trace 用 `tools/trace-diff.py` 比——**任何 1-dot 偏移都会让 PPU 列(scanline,dot)
   或 NMI 时点漂移**,这是抓重构回归的最快办法。先与「重构前的 fc 自身 trace」对齐到 0-diff,再考虑对 Mesen。

## 建议推进步骤(小步、可回退)

1. `git pull`/`git status` 对齐;通读 `ppu.rs` 的 `tick()` 全貌 + 上面列的子函数;跑一遍三档 `fc bench --profile`
   和 47/47 基线,**记录重构前数字**(本文件表格可作起点,但以你机器实测为准)。
2. (可选)给 profiler 加「PPU 核」消融档,量出 PPU 核在「其余」里的真实占比,确认投入产出。
3. **不改行为地**重组:把 `tick()` 按 scanline 阶段(visible / prerender / postrender(240) / vblank(241-260))
   拆成分发 + 各段函数;可见/预渲染段内再按 dot 区间(取数 2-257 & 321-337、sprite 评估、HBlank…)拆。
   先做**纯结构等价重排**(逻辑一字不改),每步过三道门,确认仍 0-diff、基线全绿、画面正常。
4. 结构清晰后再做**热路径优化**:消除非渲染 dot 的无谓判断、查表替代分支、减少每 dot 固定开销;
   每步重新 `fc bench --profile` 取证、过基线 + trace parity。
5. 顺带可做 L1.2(`render_pixel` 去 `SpriteUnit` 拷贝,见 `sprite_pattern_pixel(sprite: SpriteUnit)` 改 `&`)、
   L1.3(帧缓冲直出预乘 `u32` LUT)——都是低风险快赢,但**先把 L1.1 的等价重排稳住**。
6. 落地为独立 OpenSpec change(`openspec/changes/<名>/`,proposal→design→tasks),与 M0 同纪律守门。
