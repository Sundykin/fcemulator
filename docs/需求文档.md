1. 项目概述
1.1 项目定位
本项目是一套从零基于纯 Rust 开发、周期精确（Cycle-Accurate）的跨平台 FC/NES 红白机模拟器。采用「无 IO 纯逻辑核心库 + 多前端适配」的标准化架构，整体包含四大独立模块：fc-core 纯模拟器核心库、fc-cli 命令行前端、fc-gui 桌面可视化 GUI 前端、fc-mcp AI 交互服务端。
项目核心差异化创新：原生基于 MCP（Model Context Protocol）协议封装模拟器全量能力，将游戏操控、状态读取、画面捕获、单步调试、内存读写等功能标准化为 AI 可调用工具，原生支持 LLM 大模型观战、智能陪玩、自动代打、游戏逆向辅助，是兼顾复古游戏体验、硬件逆向调试、AI 智能交互的一体化 NES 开发平台。
1.2 产品愿景
- 高精度模拟器：1:1 复刻原生 FC 硬件时序、画面渲染、音频输出、按键手感，兼容 80%+ 主流商业 FC 游戏，达到入门级商用模拟器精度标准。
- 专业逆向调试平台：内置全套可视化调试工具，支持 6502 指令反汇编、硬件断点、运行追踪、PPU/APU/内存可视化查看，满足 ROM Hack、TAS 制作、嵌入式教学、FC 硬件逆向研发需求。
- AI 游戏交互底座：通过标准化 MCP 协议开放模拟器底层能力，构建 LLM 与实时游戏系统的原生交互闭环，支撑 AI 自主游戏、策略分析、场景决策、自动化测试等创新场景。
1.3 目标用户
- 普通怀旧用户：追求低延迟、高画质、纯净无广告的桌面 FC 游戏模拟器，用于日常复古游戏游玩。
- 开发研究用户：6502 单片机学习、FC 硬件原理研究、ROM 修改、TAS 速通制作、模拟器开发学习者。
- AI 研究用户：探索 LLM 实时环境交互、多模态游戏识图、智能 Agent 决策、游戏自动化交互的技术研究者与开发者。

2. 项目目标与非目标（v1.0 全新开发）
2.1 核心开发目标（In Scope）
- G1【本期开发】：从零实现周期精确的 Ricoh 2A03 CPU、2C02 PPU、2A03 内置五声道 APU，完全对齐原生硬件时序规则。
- G2【本期开发】：完整实现 iNES / NES 2.0 标准 ROM 解析，适配主流 Mapper，兼容 80%+ 主流商业 FC 游戏。
- G3【本期开发】：精准适配 NTSC / PAL / Dendy 三大区域硬件时序、帧率、分频、渲染规则。
- G4【本期开发】：实现整机全量状态快照存档（Save State）+ 游戏电池持久存档（SRAM .sav），覆盖全部硬件模块状态。
- G5【本期开发】：支持双路标准手柄、Zapper 光枪外设输入，支持自定义键位映射与按键持久化。
- G6【本期开发】：内置系统默认调色板，支持自定义 .pal 调色板文件导入、切换、实时预览。
- G7【本期开发】：搭建全套调试体系，支持指令断点、内存观察点、运行追踪、反汇编解析、PPU/APU/内存可视化调试。
- G8【本期开发】：完整交付三套可用前端：CLI 无头调试前端、egui 低延迟桌面 GUI 前端、MCP AI 服务端前端。
- G9【本期开发】：基于 cpal 实现低延迟无爆音音频输出，基于 wgpu+egui 实现 GPU 硬件加速游戏画面渲染。
- G10【本期开发】：适配官方 NES 测试 ROM 套件，可量化校验模拟器硬件准确度，满足标准化验收标准。
2.2 非开发目标（Out of Scope，v1.0 不开发）
- N1：不完整支持 FDS 磁碟机、VS. System、街机版 FC 专属外设，仅做文件头部标识解析，不实现硬件模拟逻辑。
- N2：不做全量冷门 Mapper 兼容，仅维护主流核心 Mapper 列表，小众游戏不做适配。
- N3：不实现网络联机对战、游戏视频录制导出、直播推流功能。
- N4：v1.0 不开发移动端、WASM 网页前端，架构预留扩展接口，后续可迭代开发。
- N5：不开发 NSF/NSFe 纯音乐独立播放模式。

3. 术语表
缩写
全称
详细说明
CPU
Ricoh 2A03
FC 专属 6502 变种处理器，移除 BCD 十进制运算模式，内置 APU 音频处理单元，为模拟器核心运算单元
PPU
Ricoh 2C02
FC 图形处理单元，原生输出 256×240 标准分辨率，负责图块渲染、精灵绘制、画面滚动、调色板输出、中断触发
APU
Audio Processing Unit
音频处理单元，包含双矩形波、三角波、噪声、DMC 采样共五声道，负责游戏全部音效与背景音乐输出
Mapper
卡带内存映射器
FC 卡带核心控制模块，用于切换 PRG/CHR 存储体、控制屏幕镜像、触发自定义 IRQ 中断，是游戏兼容的核心组件
PRG-ROM/RAM
程序存储/运行内存
PRG-ROM 存储游戏程序代码，PRG-RAM 为可读写内存，用于存放游戏运行数据与电池存档数据
CHR-ROM/RAM
图块存储内存
存储游戏贴图、精灵图块素材，CHR-ROM 为只读素材，CHR-RAM 支持游戏运行时动态修改贴图数据
OAM
Object Attribute Memory
精灵属性内存，共256字节，可存储64个游戏精灵的位置、尺寸、配色、翻转、优先级等属性数据
Loopy 寄存器
PPU 滚动寄存器组
由 v/t/x/w 四组寄存器组成，控制画面精细滚动、名称表切换，是 PPU 时序与画面渲染的核心控制单元
NMI/IRQ
硬件中断
NMI 为 VBlank 垂直空白中断，用于游戏帧刷新；IRQ 为通用中断，由 Mapper、APU 触发，用于游戏逻辑调度
MCP
Model Context Protocol
基于 JSON-RPC 2.0 的 LLM 工具调用协议，用于实现 AI 大模型与模拟器的双向通信、工具调用、状态交互

4. 系统整体架构
4.1 工作区目录结构（Cargo Workspace）
全新采用核心与前端完全解耦架构，fc-core 为纯逻辑、无 IO、无渲染、无前端依赖的基础库，所有前端模块统一调用核心 API，保证全端行为一致性、可测试性、可扩展性。
fc/
├── Cargo.toml          # 工作区根配置，统一依赖版本、编译参数、发布配置
├── fc-core/            # 【核心纯逻辑库】零IO、零渲染、零前端依赖，纯硬件模拟逻辑
│   └── src/
│       ├── lib.rs
│       ├── types.rs        # 全局枚举、结构体、时序规则、输入定义、中断定义
│       ├── cartridge/      # ROM解析、卡带数据解析、Mapper 适配调度
│       ├── cpu/            # 2A03 处理器指令、时序、中断、DMA 完整实现
│       ├── ppu/            # 2C02 图形流水线、寄存器、调色板、精灵、滚动逻辑
│       ├── apu/            # 五声道音频生成、混音、采样输出、中断触发
│       ├── mapper/         # Mapper 通用Trait + 主流型号具体实现
│       ├── bus.rs          # 总线地址译码、硬件周期联动、时序统一推进
│       ├── input.rs        # 手柄、光枪输入数据解析与状态管理
│       ├── control_deck.rs # 顶层统一API（所有前端唯一调用入口）
│       ├── save_state.rs   # 整机全量快照存档、电池持久存档逻辑
│       └── debug/          # 调试器、反汇编、断点、内存观察、运行追踪
├── fc-cli/             # 命令行前端：无头运行、调试输出、ROM测试、MCP服务启动
├── fc-gui/             # 桌面GUI前端：eframe+egui+wgpu+cpal 纯Rust低延迟渲染
├── fc-mcp/             # MCP AI服务端：标准化AI工具封装、协议通信、状态交互
├── nes-test-roms/      # 官方标准化测试ROM套件（用于准确度验收）
└── docs/               # 项目需求文档、设计文档、开发手册、验收规范

4.2 核心依赖边界（强设计约束）
为保障核心库高复用、高可测、高可移植性，全新定义严格的模块依赖边界，全程无违规跨层依赖：
- fc-core（纯核心层）：禁止依赖 winit、wgpu、cpal、tokio、clap 等 IO、渲染、异步、命令行库。仅依赖 serde、bincode、bitflags、bytemuck、thiserror、log 等基础工具库。天然支持桌面、测试、WASM、嵌入式等多场景复用。
- fc-cli（命令行层）：仅依赖 fc-core、fc-mcp、clap，负责无头调试、自动化测试、服务启动。
- fc-gui（桌面界面层）：仅依赖 fc-core、eframe、wgpu、winit、cpal、rfd，负责图形渲染、音频输出、用户交互、可视化调试。
- fc-mcp（AI 交互层）：仅依赖 fc-core、协议解析库，负责标准化 AI 工具封装、远程状态交互。
4.3 核心一帧数据流设计（全新标准化设计）
采用子指令级周期精确联动设计，以 PPU 周期为主时钟，CPU、APU、Mapper 严格跟随硬件比例同步推进，杜绝时序错位，完全对齐真机硬件逻辑。
4.4 核心时序设计准则（全新开发核心规范）
- 全局以 PPU 周期为唯一主时钟，NTSC 制式严格遵循 1 CPU 周期 = 3 PPU 周期；PAL 制式采用 5:16 整数分频比，杜绝时序漂移。
- 严格实现子指令级周期捕获，CPU 指令执行过程中，每一次总线读写都实时同步推进 PPU/APU 时序，不允许指令执行完成后统一空转补周期。
- 所有硬件行为、中断触发、数据读写、寄存器更新，完全遵循 NESDEV 官方硬件文档，零自定义硬编码、零游戏专属 Hack。
